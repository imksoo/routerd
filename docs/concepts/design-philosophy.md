# 設計思想

routerd は、ルーターを「設定ファイルの集まり」ではなく「状態を持つリソースの集合」として扱います。
このページでは、実装の判断基準を説明します。

## YAML を意図の中心に置きます

YAML はルーターの意図を表します。
routerd は YAML とホストの現在状態を比べ、必要な差分だけを適用します。
作業手順の記憶ではなく、読み返せる定義を重視します。

## 状態を持つ処理は専用デーモンに分けます

DHCPv6-PD、DHCPv4、PPPoE、ヘルスチェックのような処理は、タイマー、再起動復元、イベント履歴を持ちます。
これらを一度だけのコマンドに押し込むと、リース更新や障害時の観測が不安定になります。

routerd では、状態を持つ処理を小さな専用デーモンとして動かします。
デーモンは lease や state をファイルに保存し、Unix ドメインソケットで状態を公開します。
routerd 本体はその状態を読んで、下流のリソースを調整します。

## 壊れた IPv6 を LAN に出しません

DHCPv6-PD が失われているのに、古いプレフィックスの RA、AAAA、LAN アドレスを出し続けると、利用者からは「IPv6 があるように見えて通信できない」状態になります。
routerd は、委譲プレフィックスの状態を確認できないとき、下流の IPv6 提供を止める設計を取ります。

## 小さな部品をイベントでつなぎます

routerd は、ひとつの大きな手続きでルーター全体を処理するのではなく、小さな制御器をイベントでつなぎます。
たとえば、DHCPv6-PD が Bound になると、LAN アドレス、RA、DHCPv6 サーバー、DNS 応答、DS-Lite、IPv4 経路へ順に状態が伝わります。

この形にすると、どの段階で止まったかを見つけやすくなります。

## OS 差分は閉じ込めます

Ubuntu、NixOS、FreeBSD では、同じ意図でもホスト側の表現が異なります。
routerd は `pkg/platform` の機能フラグと OS 別生成器で差分を閉じ込めます。
利用者向けのリソース名は、できるだけ同じ形に保ちます。

## 互換性よりも正しい名前を優先する段階です

現在は出荷前の v1alpha1 です。
Phase 1.6 では、`DHCPv4*` と `DHCPv6*` の RFC 表記に合わせるため、古い Kind 名とバイナリ名を互換別名なしで整理しました。
この段階では、将来ずっと残る誤った名前を避けることを優先します。
