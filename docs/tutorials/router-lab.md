# Building an evaluation lab

Before you put routerd on a production network, evaluate it in an isolated lab. This page describes a reference layout for spinning up several router VMs on any hypervisor and exercising routerd across multiple operating systems.

The instructions are not tied to any specific hardware. Proxmox VE, KVM, VMware, and Hyper-V all work.

## Scenario

You want to validate the following before pointing routerd at a production segment:

- DHCPv6-PD, DS-Lite, PPPoE, NAT44, firewall, and DNS behave as expected.
- Linux, FreeBSD, and NixOS are interchangeable for the same configuration.
- A change can be reproduced safely in a sandbox before being applied to production.

A lab achieves three things: it isolates failures from production, it lets you swap operating systems freely, and it accepts the same routerd YAML you intend to deploy later.

## Hardware

Everything runs on virtual machines. No physical router is required.

- One virtualization host (Proxmox VE, KVM, VMware, or Hyper-V).
- 4 GB or more of free memory (about 512 MB per VM).
- An upstream network. To exercise DHCPv6-PD you need a residential gateway (or a simulator) that delegates IPv6 prefixes.

## Recommended VM topology

The minimum is **two VMs**; a useful coverage starts at **four to five VMs**.

| Role | Example OS | What it exercises |
| --- | --- | --- |
| Measurement host | Ubuntu / Debian | `iperf3`, `dig`, `curl`, `mtr` and friends |
| WAN-side router A | Ubuntu | `routerd-dhcpv6-client`, PPPoE, DS-Lite |
| WAN-side router B | NixOS | The NixOS module path that wraps routerd |
| WAN-side router C | FreeBSD | FreeBSD `pf`, DS-Lite, rc.d unit handling |
| LAN-side router | Ubuntu | Controller chain, DNS, firewall, NAT, `HealthCheck`, Web Console |

Connect the VMs to a shared virtual switch (VLAN trunk or untagged bridge) and keep that switch isolated from the upstream. Wire each VM with one NIC into an "upstream" vSwitch and another NIC into the "lab" vSwitch.

## Acceptance checklist

Use these checks to confirm the lab is healthy:

1. **DHCPv6-PD acquired** — `routerctl status` reports `DHCPv6PrefixDelegation` as `Bound` on every router VM.
2. **No PD overlap** — When you bring up five or more routers, the prefixes do not collide. The HGW must distribute distinct IA_PDs.
3. **IPv6 reachability** — Lab VMs can ping each other on link-local and global addresses, both directions.
4. **DS-Lite** — `routerctl describe DSLiteTunnel/<name>` reports `Up`, the matching `HealthCheck` is `Healthy`, and `curl --interface ds-lite-X http://example.com/` returns a response.
5. **NixOS path** — `nixos-rebuild test` materialises the routerd configuration and the same state survives a reboot.
6. **FreeBSD path** — `pfctl -sr` shows the rules generated by routerd; `service routerd onestatus` is active.

## Out of scope

The following depend on devices outside routerd. Do not waste lab time reproducing them:

- An HGW that does not return the AFTR option in DHCPv6 information-request — pin a static fallback in `DSLiteTunnel.spec.aftrFQDN`.
- An ISP that only honours specific IA_IDs — fix the IA_ID on the client side.
- A physical switch that drops IPv6 RA on a VLAN — bypass that switch by routing inside the same vSwitch.

These cases are catalogued under `docs/knowledge-base/`.
